# BabelDOC Web — 系统架构与设计思路

## 1. 项目概述

BabelDOC Web 是基于 [BabelDOC](https://github.com/funstory-ai/BabelDOC) 核心翻译引擎构建的全栈 Web 文档翻译平台。系统提供了完整的 PDF 文档上传、AI 翻译、结果下载、术语管理、模型配置以及后台管理能力，支持在线与离线两种部署模式。

### 1.1 核心设计目标

| 目标               | 说明                                                           |
| ------------------ | -------------------------------------------------------------- |
| **易用性**   | 拖拽上传 PDF，一键翻译，自动检测重复任务                       |
| **可扩展**   | 支持多模型配置（任何 OpenAI 兼容 API），灵活的 extra_body 透传 |
| **专业性**   | 术语表管理、协作术语库、AI 自动提取术语                        |
| **离线就绪** | 完整的离线资源检测、导出、恢复机制，适配内网/气隙环境          |
| **多用户**   | JWT 认证、RBAC 角色控制（管理员/普通用户）                     |

### 3.2 分层架构

系统采用经典的**三层架构**：

1. **API 路由层** (`api/`) — 处理 HTTP 请求，参数校验，响应构建
2. **业务服务层** (`services/`) — 翻译队列、翻译执行、资源管理
3. **数据访问层** (`models/` + `core/database.py`) — ORM 模型与数据库交互

### 3.3 异步翻译队列

翻译任务使用 `asyncio.Queue` + `asyncio.Semaphore` 实现并发控制：

```
用户提交 → Task(pending) → 入队(queued) → 获取信号量 → 执行翻译(running) → 完成/失败
                                              │
                    MAX_CONCURRENT_TRANSLATIONS │ (默认 2)
                                              │
                              asyncio.Semaphore 控制并发
```

**设计考量：**

- `MAX_QUEUE_SIZE` (默认 100) 限制排队上限，防止内存溢出
- `MAX_CONCURRENT_TRANSLATIONS` (默认 2) 限制同时运行的翻译任务数
- BabelDOC 翻译是 CPU/IO 密集型，通过 `asyncio.to_thread()` 放到线程池执行
- 进度回调：翻译过程中实时更新数据库中的 `progress` 和 `progress_message`

### 3.4 重复翻译检测

系统通过 **文件哈希 + 模型配置哈希** 实现智能去重：

```
┌──────────────────┐    ┌──────────────────────────┐
│  SHA256(文件内容)  │ +  │  StableHash(翻译配置JSON)  │  =  唯一翻译指纹
└──────────────────┘    └──────────────────────────┘
```

当用户上传文件后，系统自动检查是否存在**相同文件 + 相同翻译配置**的已完成任务：

- **存在** → 前端弹出复用对话框，用户可选择下载已有结果或重新翻译
- **不存在** → 正常排队翻译

## 6. 安全设计

### 6.1 认证与授权

| 安全措施 | 实现方式                                   |
| -------- | ------------------------------------------ |
| 密码存储 | bcrypt 哈希，不可逆                        |
| 会话管理 | JWT (HS256)，24 小时有效期                 |
| 角色控制 | `require_admin()` 依赖注入，路由级别控制 |
| 首个用户 | 自动成为管理员，确保系统可管理             |
| 用户隔离 | 普通用户仅可访问自己的任务、术语表         |

### 6.2 输入验证

- Pydantic schema 进行请求参数验证
- 文件上传限制 PDF 格式
- extra_body 字段进行 JSON 格式验证
- 防止管理员自我降级/删除最后一个管理员

## 7. 离线部署设计

### 7.1 离线资源体系

BabelDOC 运行需要多种资源（模型、字体、CMap、tiktoken 编码器）。系统提供了完整的离线资源管理流程：

```
联网环境                                     离线环境
┌──────────────────┐                    ┌──────────────────┐
│ 1. 管理员点击     │                    │ 4. 上传离线包到    │
│   「预热并导出」   │                    │    目标服务器      │
│                  │                    │                  │
│ 2. 系统自动下载   │   ════ 拷贝 ════▶  │ 5. 管理员点击      │
│    全部运行资源   │   offline.zip      │   「恢复离线包」   │
│                  │                    │                  │
│ 3. 打包为 ZIP    │                    │ 6. 自动解压到      │
│    供下载        │                    │    缓存目录        │
└──────────────────┘                    └──────────────────┘
```

### 7.2 Profile 分级

| Profile     | 包含资源                         | 使用场景             |
| ----------- | -------------------------------- | -------------------- |
| `full`    | Models + Fonts + CMap + Tiktoken | 严格离线环境（推荐） |
| `core`    | Models + Fonts + Tiktoken        | 平衡模式             |
| `minimal` | Models + Tiktoken                | 最小化启动检查       |

### 8.2 自动术语提取

翻译时启用「自动提取术语」，BabelDOC 在翻译过程中会自动识别并提取专业术语对照，翻译完成后用户可以一键保存为新的术语表，用于后续翻译任务。

## 10. 可扩展性设计

### 10.1 模型扩展

系统支持任何兼容 OpenAI Chat Completions API 的模型服务：

- 云端模型：OpenAI、DeepSeek、智谱等
- 本地模型：通过 vLLM / Ollama / llama.cpp 等提供的 OpenAI 兼容端点
- 每个模型可独立配置：API Key、Base URL、Temperature、Reasoning Level、extra_body

### 10.2 extra_body 透传机制

```
模型级 extra_body          任务级 extra_body
{"stream": false}    +     {"reasoning": {"effort": "high"}}
        │                           │
        └─── 合并 (任务级覆盖) ──────┘
                    │
                    ▼
         最终传递给翻译 API
```

这使得用户可以在不修改模型配置的情况下，为特定翻译任务传递额外参数。
